Pull & Conflicts

ما الذي قمنا به في الدرس السابق؟

في الدرس السابق قمنا بعمل إستنساخ (Clone) لمشروع من مستودع على موقع الـ GitHub.

ما الذي سنستفيدة من هذا الدرس؟

إن إختيار طريقة العمل بأن نكون جميعنا (contributors) سيساعدك على تجربة العمل كفريق وحل التعارضات (conflicts) عند مواجهتها.

سحب التعديلات التي قام بها المطورين الأخرين

تذكر في كل مرة قبل أن تقوم بالبدء بالعمل على تعديلاتك يجب أن تقوم بتحديث النسخة التي لديك بالقيام بسحب الملفات من المستودع الذي على GitHub فهناك أشخاص أخرون يعملون معك وربما قاموا برفع تعديلات جديدة على المشروع.

وللقيام بذلك سيتطلب منك عمل جلب للملفات ومن ثم دمجها مع الملفات الموجودة على مستودعك المحلي ويمكنك القيام بذلك بطريقتين:

الطريقة الأولى:

جلب التعديلات عبر الأمر git fetch ومن ثم دمجها بإستخدام الأمر git merge

الأمر git fetch

git fetch

أو الأمر التالي فلن يختلف ذلك في درسنا الحالي

git fetch origin master

هذا الأمر يقوم بتنزيل وجلب التعديلات من على المستودع البعيد إلى المستودع المحلي فقط.

تم تنزيل التحديثات والأن لنقم بتشييك حالة المستودع الخاص بنا بعد قيامنا بذلك:

لاحظ أن المستودع المحلي متاخر عن المستودع البعيد بـ commit واحدة. ما يعني بأنه أحد أعضاء الفريق قام بعمل commit جديدة وقام برفعها للمستودع.

أيضاً بعد قيامك بتطبيق الأمر git fetch ستلاحظ بأنه لم يتم إحداث أي تغيير على الملفات التي تعمل عليها ولكن أصبح لديك في مستودعك المحلي أخر تحديث للملفات بينما انت ما تزال تعمل على الفرع الخاص بك.

راجع الفيديو التالي حتى الدقيقة 0:32 فقط:

كما شاهدت في الفيديو أعلاة فالأمر git fetch قام بتنزيل التحديثات من المستودع البعيد ولكن مازلنا نعمل على الفرع الخاص بنا بالمستودع المحلي master كما بالصورة التالية:

الأن سأقوم بإستخدام الأمر git diff لمشاهدة الفرق بين ملفاتي وبين التحديثات الجديدة:

لاحظ في الصورة أعلاة أن الفرع الذي أعمل عليه بإسم master وأن الفرع الخاص بتتبع التعديلات على المستودع البعيد هو origin/master وأريد ان أعرف الفرق بينهما لهذا سأقوم بكتابتهما كالتالي:

git diff master origin/master

النتيجة:

من النتيجة أعلاة أستطيع أن أعرف بأن الـ commit التي مستودعي المحلي متاخر بها عن المستودع البعيد هي في ملف الـ index.html حيث أنه يتم الإشارة إليها بعلامة + على اليسار ولون الكود الخاص بالتعديل بالون الأخضر.

بإمكانك ايضاً مشاهد الإختلافات في الملف كامل بإستخدام الأمر التالي:

git diff origin/master

لنلقي نظرة على ملف الـ index.html على مستودعي المحلي:

من خلال المقارنة الأن أصبحت أعرف أنه اذا ما سأقوم بدمج الملف الخاص مع التعديلات الجديدة فإنه سيتم إضافتها مكان السهم الأخضر في الصورة أعلاة ولا يوجد أي تعارض لدي حيث أن الدمج لن يسبب أي مشاكل بالكود الخاص بي. لذا سأنتقل لأمر الدمج.

الأمر git merge

كما سبق وقلنا أن الأمر git fetch يقوم بجلب التعديلات ولكن لا يقوم بعمل أي تغيير على العمل الخاص بك وحتى يتم دمج التعديلات تحتاج لإستخدام الأمر merge إرجع للفيديو السابق وشاهد بقيتة.

سأقوم بتطبيق الأمر كالتالي:

git merge origin/master

النتيجة:

لاحظ أنه تم القيام بعملية الدمج وإضافة 7 أسطر جديدة في ملف index.html كما هو موضع في واجهة الأوامر.

لنلقي نظرة على الملف الخاص بي:

تم إضافة الأسطر الجديدة دون حصول أي تعارض.

الطريقة الثانية:

نستطيع القيام بما سبق بطريقة أخرى عن طريق جلب التعديلات من المستودع البعيد ودمجها في نفس الوقت بإستخدام الأمر pull.

الأمر git pull

هذا الأمر يقوم بسحب التعديلات ودمجها في نفس الوقت أي انه يقوم بعمل الأمر git fetch ومن ثم يتبعة بعمل الأمر git merge.

شاهد الصورة التالية لتعرف الفرق بين الطريقة الأولى والطريقة الثانية:

لنقم بتطبيق الأمر git pull:

لنفترض أن أحد المطورين قام بعمل تعديل للمستودع وأريد أن أقوم بجلب هذه التعديلات ودمجها مع ملفات مستودعي المحلي, سأقوم بكتابة الكود كالتالي:

git pull origin master

النتيجة:

لاحظ في واجهة الأوامر تم إخباري بأنه تم جلب التعديلات ودمجها عبر إضافة 6 أسطر جديدة في ملف الـ index.html

متى تستخدم git fetch ومتى تستخدم git pull ؟

الأمر fetch: يقوم بجلب التعديلات من المستودع البعيد و من الفرع المستهدف وتخزينها في المستودع المحلي تحديداً في ما يسمى Remote Tracking Branches دون دمج تلك التعديلات بالفرع الذي يتم العمل علية مما يسمح لك بمراجعة الإختلافات قبل القيام بعملية الدمج.

الأمر pull: يقوم بجلب التعديلات من المستودع البعيد ومن ثم يقوم بدمج التعديلات بشكل مباشر مع الفرع الذي جاري العمل علية في المستودع المحلي بدون السماح لك بمراجعة الإختلافات مما قد يسبب حصول تعارض في الأكواد.

يعتبر إستخدام الأمر git fetch أكثر أمان لتجنب حصول تعارض في الأكود مما يسمح لك بمراجعة التعديلات والفرق بين ملفاتك والملفات التي تم جلبها من المستودع البعيد لتتجتب حصول فوضى عند القيام بعملية الدمج.

غالباً يتم إستخدام الأمر git fetch عندما يكون لديك على مستودعك المحلي commits غير موجودة في المستودع البعيد وكذلك يوجد commits في المستودع البعيد وغير موجودة على المستودع المحلي.

مثال:

لنفترض أنني بدأت العمل على المشروع وقمت بعمل جلب التعديلات الأخيرة قبل أن أبداء عبر الأمر git pull.

ومن ثم بدأت بالعمل وقمت بعمل commit وفي نفس الوقت يوجد مساهم أخر في المشروع يقوم بعمل تعديلات وقام بعمل commit للمستودع البعيد حيث أنه الأن أصبح لدي على مستودعي المحلي commit لم أقم برفعة للمستودع البعيد بعد وكذلك يوجد commit في المستودع البعيد لم أقم بجلبة بعد.

لنفترض أنني أريد أن أقوم برفع تعديلاتي للمستودع البعيد دون أن أقوم بالتأكد من وجود أي تعديلات سبقني أحد المطورين بوضعها كالتالي:

نلاحظ انه تم رفض رفع عملية الرفع بسبب وجود تعديلات على المستودع البعيد ليست على مستودعي المحلي.

في هذه الحالة يجب علي أن أقوم بجلب تلك التعديلات للمستودع المحلي وهناك إحتمالية أنني سأواجة تعارض في الأكواد إذا ما قام ذلك المساهم بعمل تعديلات في نفس الأسطر التي قمت أنا بالعمل عليها وحينها سيكون علي أن اقوم بحل هذا التعارض بشكل يدوي.

لنقم بتجربة الأمر git pull كالتالي:

git pull origin master

النتيجة:

تم جلب التعديلات ولكن لم يعمل الأمر pull كما يجب فقد فشلت عملية الدمج بسبب وجود تعارض بين تعديلاتي وبين تعديلات المساهم الذي قام بعمل commit قبلي.

سأقوم الأن بفتح ملف الـ index.html حتى أقوم بحل التعارض بين الأكواد :

تلاحظ في الصورة أعلاة انه تم وضع فواصل والتي قمت بتظليلها باللون الأصفر

<<<<<<<< HEAD

الأن سأحتاج لأن أقوم بإصلاح التعارض بشكل يدوي ومن ثم مسح الفواصل حيث سيصبح الكود الخاص بي كالتالي:

بعد أن قمت بترتيب الكود وإصلاح التعارض سأقوم بحفظ الملف index.html ومن ثم سأقوم بعمل staging و commit للتعديل كما سبق وشرحنا في الدروس السابقة كالتالي:

git add index.html

ثم أتبعة بالأمر التالي:

git commit -m "Fix fourth question conflict"

النتيجة:

الأن سأقوم برفع التعديلات للمستودع البعيد بالأمر التالي:

git push origin master

النتيجة:

تم الرفع بنجاح بعد أن تعاملنا مع التعارض الذي حصل لنا.

مهمة اليوم

في الدرس السابق قمنا بعمل clone لمشروع من مستودع على موقع الـ GitHub حيث يفترض أن يكون لديك الملفات كالتالي:

بعد أن قمت بعمل إستنساخ للمشروع وبعد أن تم إضافتك كمساهم فية (contributor) أصبح لديك إمكانية القيام بعمل التعديلات على المشروع وحفظها ومن ثم عمل push للملفات من المستودع الذي على جهازك لمستودع الـ GitHub بشكل مباشر.

المهمة كالتالي:

1- قم قبل كل شي بعمل تحديث للنسخة التي لديك بإستخدام الأمر الخاص بذلك كما شرحنا في الدرس أعلاة.

سحب الملفات من المستودع البعيد قبل البدء بالعمل يضمن لك حصولك على أحدث نسخة من الموقع بعد قيام أعضاء الفريق بالتعديل عليها فلا تنسى أن تبدأ بهذا الأمر دائماً.

2- قم بفتح ملف index.html ستجد التالي:

سيقوم كل واحد مننا بالإجابة على السؤال السابق ومن ثم يقوم بوضع سؤال جديد للشخص الذي يلية فكيف سنقوم بذلك؟

قم بفتح ملف الـ index.html بإستخدام محرر الأكواد وإتبع الخطوات التالية:

ستجد الـ Object التالي:

{
number: "1",
lang: "en",
answer: "Hey there, Let's start with the first question.",
question: "If you had one wish, what would you wish for?",
image: "robot.gif"
},

قم بنسخة و لصقة ومن ثم قم بالتالي:
1- number رقم السؤال
2- lang إستخدم en اذا كنت ستكتب بالإنجليزية او ar اذا كنت ستكتب بالعربية
3- answer أكتب إجابتك عن السؤال السابق
4- question أكتب سؤالك للشخص التالي
5- image قم بوضع صورتك في مجلد images ومن ثم أكتب إسمها وصيغتها في حال لا تريد وضع صورة اترك منطقة إسم الصورة فارغ وسيتم إضافة الصورة الإفتراضية

حيث يفترض أن يكون شكل الكود السابق كالتالي بعد تعديلك:

var data = [

 {
   number: "1",
   lang: "en",
   answer: "Hey there, Let's start with the first question.",
   question: "If you had one wish, what would you wish for?",
   image: "robot.gif"
 },

 {
   number: "2",
   lang: "ar",
   answer: "هنا جوابي عن السؤال السابق",
   question: "هنا سؤالي للشخص التالي",
   image: ""
 },

];

النتيجة:

3- ستقوم بحفظ تعديلك على المستودع المحلي وستكتب رسالة الـ commit بإستخدام رقم السؤال الذي قمت بالإجابة عليه كالتالي:

git commit -m "Answer question 1"

4- قم برفع التعديل للمستودع البعيد على GitHub

5- في حال سبقك شخص أخر برفع الإجابة الخاصة بنفس السؤال عليك أن تقوم بتحديث ملفاتك والإجابة على السؤال الجديد وكذلك التعامل مع التعارض في الأكواد إن وجد ومن ثم إعادة رفع التعديل.

6- بإمكانك إعادة التطبيق أكثر من مرة بنفس الطريقة فالمرجوا هو أن تواجهك المشاكل وتعارض الأكواد حتى تستطيع تقوم بالتعامل معها ويتحقق الهدف من هذا الدرس.
